# Claude Code Instructions - Documentation Project

## Project Overview
This is a documentation project. Focus on clarity, consistency, and maintainability of written content.

## Documentation Standards

### Writing Style
- Use clear, concise language
- Write in active voice
- Define technical terms on first use
- Use consistent terminology throughout

### File Organization
- Place high-level docs in `/docs`
- Place step-by-step guides in `/guides`
- Place technical specs in `/reference`
- Keep assets organized in `/assets`

### Markdown Conventions
- Use ATX-style headers (# Header)
- Include table of contents for long documents
- Use code blocks with language specification
- Add alt text for diagrams

## Tasks Guidelines

### When Creating New Documentation
- Follow the existing structure and naming conventions
- Add entries to relevant index files
- Cross-reference related documents
- Keep file names lowercase with hyphens

### When Editing Documentation
- Maintain consistent formatting
- Update all related cross-references
- Preserve existing voice and tone
- Check for broken links

### Before Committing
- Verify all links work
- Check markdown syntax
- Ensure consistent formatting
- Review for clarity and completeness
- Validate YAML frontmatter follows tagging standard

## Repository Organization

### Meta-Documentation Location
All repository reorganization plans, status documents, and structural changes should be documented in:
`/docs/repository-organization/`

This includes:
- Directory structure change proposals
- Content consolidation and migration plans
- Implementation status tracking
- Historical reorganization docs (archived)

**Current Plan:** `docs/repository-organization/2025-01-client-onboarding-reorganization.md`

### Directory Structure Principles
- **Minimize subdirectories**: Only create subdirectories when there's a clear functional distinction
- **Flat is better**: Keep files at directory root level when possible
- **No empty placeholders**: Don't create directories until content exists
- **Consistent naming**: Use lowercase-with-hyphens pattern

### Client Onboarding Documentation
**Primary Location:** `04-operations/sops/client-onboarding/`

This is the single source of truth for client onboarding procedures. Supporting technical documentation is in `05-platform/client-onboarding/`.

## Tagging Standards

All documents must include properly formatted YAML frontmatter with appropriate tags.

### Key Tagging Principles
- Use 2-6 tags per document
- Follow established tag vocabulary in [SC Tagging Standard](08-reference/standards/sc-tagging-standard.md)
- Don't duplicate directory structure with tags
- Only create new tags if they serve >5% of repository content
- Use hierarchical tags for clients and platforms (e.g., `client/upscale-legal`)

### Required YAML Frontmatter
```yaml
---
title: Document Title
version: 1.0
author: Symphony Core Systems Team
last_updated: YYYY-MM-DD
category: [guide, sop, standard, etc.]
tags: [tag1, tag2, tag3]
status: draft|active|review|archived|deprecated
audience: internal-team|client-facing|all-teams
---
```

### External Tool Integration
External tools and repositories that read this repository can:
- Parse YAML frontmatter to extract metadata
- Filter documents by tags using standard YAML parsers
- Generate documentation indexes and navigation
- Automate workflows based on document status and tags
- See [SC Tagging Standard](08-reference/standards/sc-tagging-standard.md) for complete tag vocabulary

## Important Reminders
- NEVER create files unless explicitly requested
- ALWAYS prefer editing existing files
- Focus on documentation quality over quantity
- Prioritize user comprehension
